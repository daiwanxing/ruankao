# 第二章 信息系统工程

## 2.1 信息系统

### 一、教学目标

掌握信息系统的基本概念、功能、物理结构、通用结构以及建设原则，理解集中式与分布式系统的区别。

### 二、核心讲解

#### 1. 信息系统基本概念

**定义**：信息系统是一组相互关联的元素或组件，它们收集（输入）、操作（处理）、存储和传播（输出）数据与信息，并提供满足目标的反馈机制。

**五个基本功能**：
1. 输入
2. 存储
3. 处理
4. 输出
5. 控制

**诺兰六阶段模型**：
初始阶段 → 传播阶段 → 控制阶段 → 集成阶段 → 数据管理阶段 → 成熟阶段

#### 2. 信息系统的物理结构

**集中式结构**：
- **优点**：
  - 资源集中，便于管理
  - 资源利用率较高
- **缺点**：
  - 维护与管理困难
  - 不利于调动用户积极性
  - 资源过于集中，系统脆弱，易瘫痪

**分布式结构**：
- **优点**：
  - 可根据应用需求配置资源
  - 提高应变能力
  - 系统扩展方便
  - 安全性好
  - 节点故障不会导致整个系统瘫痪
- **缺点**：
  - 资源分散
  - 系统管理标准不易统一
  - 协调困难
- **特性**：分布性、对等性、并发性、缺乏全局时钟、故障多样

#### 3. 信息系统的通用结构（自底向上）

1. **机房基础设施**：机房基础环境、安防系统、电气系统、精密空调系统、环境检测系统、消防系统
2. **物理资源**：网络、服务器、存储、终端、外设等硬件
3. **虚拟资源**：网络资源、计算资源、存储资源等
4. **平台资源**：支撑应用系统运行的基础软件（操作系统、数据库、中间件等）
5. **应用**：面向各类应用的软件系统（财务软件、HR软件、OA软件、监控软件、流程管理软件、安全分析软件等）
6. **数据**：业务数据、运维数据、安全数据等

#### 4. 信息系统建设原则

1. **高层管理人员介入原则**：CIO深度介入信息系统开发建设以及运行
2. **用户参与开发原则**：
   - 有确定的范围
   - 参与全过程
   - 深度参与
3. **自顶向下规划原则**
4. **工程化原则**
5. **其他原则**：创新性、整体性、发展性、经济性等

### 三、通俗例子

**集中式例子**：
就像早期的银行系统，所有数据和业务都集中在总行处理，分行只是终端。这种方式便于统一管理，但一旦总行系统故障，整个银行业务都会瘫痪。

**分布式例子**：
就像现在的银行系统，每个分行都有自己的业务处理能力，客户可以在任意分行办理业务。即使某个分行系统故障，其他分行仍可正常工作。

**信息系统五功能例子**：
就像一个餐厅管理系统：
- **输入**：服务员录入菜品信息、顾客点菜
- **存储**：订单信息、菜品库存存储在数据库
- **处理**：计算总价、扣减库存、统计销售
- **输出**：打印小票、显示账单、生成报表
- **控制**：权限管理、库存预警、数据备份

### 四、知识点速记

1. **信息系统五功能**：输入、存储、处理、输出、控制
2. **诺兰六阶段**：初始→传播→控制→集成→数据管理→成熟
3. **集中式优缺点**：优点（便于管理、利用率高）；缺点（维护困难、脆弱易瘫痪）
4. **分布式优缺点**：优点（灵活、扩展性好、安全性高）；缺点（分散、协调困难）
5. **通用结构六层**：机房基础→物理资源→虚拟资源→平台资源→应用→数据
6. **建设四原则**：高层介入、用户参与、自顶向下、工程化

## 2.2 系统工程

### 一、教学目标

掌握系统工程的主要方法（霍尔三维结构、切克兰德方法、并行工程等）、系统工程生命周期以及信息系统生命周期。

### 二、核心讲解

#### 1. 系统工程方法

**(1) 霍尔三维结构**
- **时间维**：规划、制定方案、研制、生产、安装、运行、更新（七个阶段）
- **逻辑维**：问题确定、目标确定、系统综合、系统分析、方案选择、评价、决策、实施计划（七个步骤）
- **知识维**：工程、医学、建筑、商业、法律、管理、社会科学、艺术（各种知识和技能）

**(2) 切克兰德方法**
- **核心**：不是"最优化"，而是"比较"与"探寻"
- **工作过程七个步骤**：认识问题 → 初步定义 → 建立概念模型 → 比较及探寻 → 选择 → 设计与实施 → 评估与反馈

**(3) 并行工程方法**
- **要点**：
  1. 在产品设计开发期间将概念设计、结构设计、工艺设计、最终需求等结合起来
  2. 各项工作由与此相关的项目小组完成
  3. 依据适当的信息系统工具，反馈与协调整个项目的进行

**(4) 综合集成法**
- **分类**：简单系统和巨系统
- **开放的复杂巨系统主要性质**：开放性、复杂性、进化与涌现性、层次性、巨量性

**(5) WSR系统方法**
- **物理**：主要应用自然科学中的各种科学方法
- **事理**：使用各种运筹学、系统工程、管理科学、控制论和数学方法
- **人理**：可以细分为关系、感情、习惯、知识、利益等

#### 2. 系统工程生命周期

**七个阶段**：
1. 探索性研究阶段
2. 概念阶段
3. 开发阶段
4. 生产阶段
5. 使用阶段
6. 保障阶段
7. 退役阶段

**生命周期方法**：
- **计划驱动方法**：整个过程始终遵守规定流程的系统化方法
- **渐进迭代式开发**：适用于较小的、不太复杂的系统，重点在于灵活性
- **精益开发**：动态的、知识驱动的、以客户为中心
- **敏捷开发**：关键目标在于灵活性

**信息系统生命周期（五个阶段）**：
系统规划 → 系统分析 → 系统设计 → 系统实现 → 系统运行与评价

### 三、通俗例子

**霍尔三维结构例子**：
就像盖房子：
- **时间维**：规划（什么时候开始）→ 制定方案（设计图纸）→ 研制（准备材料）→ 生产（建造）→ 安装（装修）→ 运行（入住）→ 更新（翻新）
- **逻辑维**：确定要盖什么房子 → 确定目标 → 综合各方面需求 → 分析可行性 → 选择方案 → 评价效果 → 制定实施计划
- **知识维**：需要建筑学、工程学、管理学、艺术学等各种知识

**并行工程例子**：
传统汽车制造是按顺序进行（设计→制造→测试→改进），并行工程则是在设计阶段就同时考虑制造工艺、装配、维护等，提前发现问题，减少后期修改。

### 四、知识点速记

1. **霍尔三维结构**：时间维（7阶段）、逻辑维（7步骤）、知识维（多种知识）
2. **切克兰德方法核心**："比较"与"探寻"
3. **并行工程三要点**：设计阶段结合、各项目小组完成、反馈协调
4. **系统工程七阶段**：探索→概念→开发→生产→使用→保障→退役
5. **生命周期四方法**：计划驱动、渐进迭代、精益、敏捷
6. **信息系统五阶段**：规划→分析→设计→实现→运行与评价

## 2.3 软件工程

### 一、教学目标

掌握软件架构、需求工程、软件设计、软件实现等软件工程的核心概念和方法。

### 二、核心讲解

#### 1. 软件架构

**核心问题**：能否达到架构级的软件复用

**软件架构风格**：

| 风格名称 | 说明 |
|----------|------|
| 数据流风格 | 批处理序列、管道/过滤器 |
| 调用/返回风格 | 主程序/子程序、数据抽象和面向对象、层次结构 |
| 独立构件风格 | 进程通信、事件驱动的系统 |
| 虚拟机风格 | 解释器、基于规则的系统 |
| 仓库风格 | 数据库系统、黑板系统、超文本系统 |

**架构评估**：
- **基于调查问卷（或检查表）的方式**
- **基于场景的方式**（最常用）
- **基于度量的方式**

**基于场景的方式包括**：
1. 架构权衡分析法（ATAM）
2. 软件架构分析法（SAAM）
3. 成本效益分析法

#### 2. 需求的层次

| 需求类型 | 含义 |
|----------|------|
| 常规需求 | 用户认为系统应该做到的功能或性能，实现越多用户会越满意 |
| 期望需求 | 用户想当然认为系统应具备的功能或性能，未实现会让用户不满意 |
| 意外需求 | 用户要求范围外的功能，实现用户会更高兴，但不实现也不影响购买决策 |

#### 3. 需求获取与分析

**需求获取方法**：
用户访谈、问卷调查、采样、会议讨论法、界面原型法、情节串联板、联合需求计划等

**需求分析**：将提炼、分析和审查已经获取到的需求

**好的需求特性**：无二义性、完整性、一致性、可测试性、确定性、可跟踪性、正确性、必要性

**(1) 结构化分析（SA方法）**
- **核心**：数据字典
- **三个层次的模型**：
  - 行为模型：状态转换图（STD）
  - 功能模型：数据流图（DFD）
  - 数据模型：实体联系图（E-R图）

**(2) 面向对象分析（OOA方法）**
- **基本任务**：运用面向对象方法，对问题域进行分析和理解，找出所需的类和对象
- **OOA模型**：
  - 用例模型（分析阶段的核心工作）
  - 分析模型（静态模型+动态模型）

#### 4. 软件设计

**结构化设计（SD）**：
- **基本思想**：将软件设计成由相对独立且具有单一功能的模块组成的结构
- **两个阶段**：概要设计、详细设计
- **基本原则**：高内聚，低耦合
- **内聚**：模块内部各成分之间的联系程度
- **耦合**：模块之间联系的程度

**面向对象设计（OOD）**：
- **基本思想**：抽象、封装和可扩展性
- **可扩展性**：通过继承和多态来实现
- **主要任务**：对类和对象进行设计（最重要的组成部分）
- **核心问题**：如何同时提高软件的可维护性和可复用性

**设计模式**：

| 分类方式 | 类型 | 说明 |
|----------|------|------|
| 处理范围不同 | 类模式 | 处理类和子类之间的关系（静态关系） |
| | 对象模式 | 处理对象之间的关系（动态性） |
| 用途不同 | 创建型模式 | 用于创建对象（工厂方法、抽象工厂、原型、单例、建造者） |
| | 结构型模式 | 处理类或对象的组合（适配器、桥接、组合、装饰、外观、享元、代理） |
| | 行为型模式 | 描述类或对象的交互以及职责的分配（职责链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者） |

#### 5. 软件实现

**软件编码**：
- **程序设计风格**：源程序文档化、数据说明、语句结构、输入/输出方法
- **程序复杂性度量**：定量度量软件的性质
- **编码效率**：程序效率、算法效率、存储效率、I/O效率

**软件测试**：

| 测试方法 | 分类 | 说明 |
|----------|------|------|
| 静态测试 | - | 被测试程序不在机器上运行，采用人工检测和计算机辅助静态分析手段检测 |
| | 对文档的静态测试 | 以检查单的形式进行 |
| | 对代码的静态测试 | 桌前检查、代码走查、代码审查、静态分析工具等 |
| 动态测试 | 白盒测试（结构测试） | 测试人员对软件内部结构和实现细节有详细了解，检查程序内部逻辑和结构 |
| | 黑盒测试（功能测试） | 测试人员不关心软件内部的实现细节，只关注程序的输入和输出结果 |

### 三、通俗例子

**架构风格例子**：
- **管道/过滤器**：像流水线，每个工人（过滤器）完成特定任务，原料（数据）在流水线（管道）中流动
- **层次结构**：像公司组织架构，董事长→总经理→部门经理→员工，层层调用
- **事件驱动**：像电梯系统，按下按钮（触发事件）→系统响应→执行相应操作

**需求层次例子**：
- **常规需求**：手机能打电话、发短信
- **期望需求**：手机电池能用一天、屏幕清晰
- **意外需求**：手机有无线充电功能（用户没要求但很惊喜）

**高内聚低耦合例子**：
- **高内聚**：一个模块只做一件事，如"用户登录模块"只负责登录，不负责注册、密码修改等
- **低耦合**：模块间依赖尽量少，如"订单模块"只需要知道"用户ID"，不需要知道用户的详细信息

### 四、知识点速记

1. **架构五风格**：数据流、调用/返回、独立构件、虚拟机、仓库
2. **需求三层次**：常规（满意）、期望（不满意）、意外（惊喜）
3. **SA三模型**：STD（行为）、DFD（功能）、E-R图（数据）
4. **SD原则**：高内聚，低耦合
5. **OOD三思想**：抽象、封装、可扩展性（继承+多态）
6. **设计模式三分类**：创建型、结构型、行为型
7. **测试两方法**：静态（文档+代码）、动态（白盒+黑盒）

## 2.4 部署交付

### 一、教学目标

掌握持续交付、持续部署、容器技术以及部署策略（蓝绿部署、金丝雀部署）。

### 二、核心讲解

#### 1. 持续交付

**定义**：完全自动化的过程，当业务开发完成的时候，可以做到一键部署

**主要体现**：
1. **需求阶段**：抛弃传统需求文档，使用便于开发人员理解的用户故事
2. **开发测试阶段**：做到持续集成，让测试人员尽早进入项目开始测试
3. **运维阶段**：打通开发和运维之间的通路，保持开发环境和运维环境的统一

#### 2. 持续部署

**常用方案**：
- Kubernetes + Docker
- Matrix系统

**完整的镜像部署（三个环节）**：
1. **Build**：将软件编译形成RPM包或Jar包
2. **Ship**：将所需的第三方依赖和第三方插件安装到环境中
3. **Run**：在不同的地方启动整套环境

#### 3. 部署策略

**(1) 蓝绿部署**
- **概念**：部署时准备新旧两个部署版本，通过域名解析切换的方式将用户使用环境切换到新版本
- **优势**：当出现问题时，可以快速地将用户环境切回旧版本

**(2) 金丝雀部署**
- **概念**：当有新版本发布时，先让少量用户使用新版本，观察新版本是否存在问题
- **优势**：如果出现问题，就及时处理并重新发布；如果一切正常，就稳步地将新版本适配给所有的用户

### 三、通俗例子

**持续交付例子**：
就像自动化生产线，从原材料（需求）输入，到产品（软件）输出，全程自动化，不需要人工干预。

**蓝绿部署例子**：
就像双车道换道，有新车道（蓝）和旧车道（绿）同时存在，车辆先在新车道试运行，确认没问题后，通过红绿灯（域名解析）将所有车辆切换到新车道。如果新车道有问题，立即切回旧车道。

**金丝雀部署例子**：
就像煤矿工人下井前，先放一只金丝雀进去探测，如果金丝雀没事，人再进去。先让少量用户使用新功能，观察没问题后再推广给所有用户。

### 四、知识点速记

1. **持续交付三阶段**：需求（用户故事）、开发测试（持续集成）、运维（打通通路）
2. **持续部署三环节**：Build（编译）→ Ship（安装依赖）→ Run（启动环境）
3. **部署两策略**：蓝绿部署（双版本切换）、金丝雀部署（小范围测试）

## 2.5 数据工程

### 一、教学目标

掌握数据建模、数据运维、数据质量、数据利用以及数据库安全等数据工程的核心概念和方法。

### 二、核心讲解

#### 1. 数据建模

**概念模型（信息模型）**：
- 按用户的观点来对数据和信息建模
- 把现实世界中的客观对象抽象为某一种信息结构
- 不依赖于具体的计算机系统，也不对应某个具体的DBMS

**逻辑模型**：
- 在概念模型的基础上确定模型的数据结构
- 主要的数据结构：层次模型、网状模型、关系模型、面向对象模型、对象关系模型
- **关系模型**成为目前最重要的一种逻辑数据模型
- **关系模型数据操作**：查询、插入、删除、更新数据
- **关系完整性约束**：
  - 实体完整性
  - 参照完整性
  - 用户定义的完整性

**物理模型**：
- 在逻辑数据模型的基础上，考虑各种具体的技术实现因素
- 进行数据库体系结构设计
- 真正实现数据在数据库中的存放

**数据建模过程**：
1. 数据需求分析（采用数据流图作为工具）
2. 概念模型设计
3. 逻辑模型设计
4. 物理模型设计

#### 2. 数据标准化

**主要内容**：元数据标准化、数据元标准化、数据模式标准化、数据分类与编码标准化

**四个阶段**：
1. **确定数据需求**：产生数据需求及相关的元数据、域值等文件
2. **制定数据标准**：处理"确定数据需求"阶段提出的数据需求，产生供审查和批准的成套建议
3. **批准数据标准**：数据管理机构对提交的数据标准建议进行审查
4. **实施数据标准**：在各信息系统中实施和改进已批准的数据标准

#### 3. 数据运维

**存储介质**：磁带、光盘、**备份磁盘（三种）

结构**：
1. DAS备份结构
2. 基于LAN的备份结构
3. LANFREE备份结构
4. SERVER-FREE备份结构

**备份策略**：
1. **完全备份**
2. **差分备份**
3. **增量备份**

**数据容灾**：
- **两个主要指标**：
  - **RPO（恢复点目标）**：当灾难发生时允许丢失的数据量
  - **RTO（恢复时间目标）**：系统恢复的时间
- **类型**：
  - **应用容灾**：保证应用服务的完整、可靠和安全
  - **数据容灾**：保证用户数据的高可用性

#### 4. 数据质量评价与控制

**数据质量元素**：
- **数据质量定量元素**
- **数据质量非定量元素**

**数据质量评价过程**：
1. 确定适用的数据质量定量元素及数据质量范围
2. 确定数据质量度量方法
3. 选择并使用数据质量评价方法
4. 确定数据质量结果
5. 报告数据质量结果（定量的 + 通过或不通过）

**数据清理的三个步骤**：
1. **数据分析**：从数据中发现控制数据的一般规则
2. **数据检测**：根据预定义的清理规则及相关数据清理算法，检测数据是否正确
3. **数据修正**：手工或自动地修正检测到的错误数据或重复的记录

#### 5. 数据利用与开发

**数据挖掘**：
- **目标**：发现隐藏于数据之后的规律或数据间的关系，从而服务于决策
- **主要任务**：数据总结、关联分析、分类和预测、聚类分析、孤立点分析
- **流程**：确定分析对象 → 数据准备 → 数据挖掘 → 结果评估与结果应用（5个阶段）
- **专业人员**：业务分析人员、数据挖掘人员、数据管理人员

**数据服务**：
1. **数据目录服务**：快捷发现和定位所需数据资源
2. **数据查询与浏览及下载服务**：网上数据共享服务的重要方式
3. **数据分发服务**：数据生产者通过各种方式将数据传送到用户的过程

#### 6. 数据库安全

**数据库安全威胁**：

| 维度 | 威胁 | 表现方式 | 安全后果 |
|------|------|----------|----------|
| 自然或意外灾害 | - | 地震、水灾、火灾等 | 完整性破坏、拒绝服务 |
| 系统软硬件中的错误 | - | 系统软硬件中错误 | 非授权信息泄露、数据修改、拒绝服务 |
| 人为错误 | - | 无意地违反安全策略 | 非授权信息泄露、数据修改、拒绝服务 |
| 有意 | 授权用户 | 滥用自己的特权造成威胁 | 非授权信息泄露、数据修改、拒绝服务 |
| 有意 | 恶意代理 | 病毒、特洛伊木马和后门 | 非授权信息泄露、数据修改、拒绝服务 |

**数据库安全机制**：用户的身份认证、存取控制、数据库加密、数据审计、推理控制

### 三、通俗例子

**概念模型例子**：
就像画房子的设计图，只关注房间布局、功能分区，不关心用砖还是用钢筋建造。

**逻辑模型例子**：
就像确定房子用框架结构还是砖混结构，选择具体的建筑方式。

**物理模型例子**：
就像具体施工，确定用多少号水泥、什么规格钢筋、多厚的墙体。

**数据容灾例子**：
就像买房，RPO是能接受多大损失（如房子没了但地还在），RTO是多久能重新有房住（如租房过渡一段时间）。

**数据挖掘例子**：
超市发现买尿布的人通常也会买啤酒，这就是通过分析购物数据发现的关联规律，用于商品摆放和促销。

3. **数据标准化四阶段**：确定需求→制定标准→批准标准→实施标准
4. **备份三策略**：完全、差分、增量
5. **容灾两指标**：RPO（数据丢失量）、RTO（恢复时间）
